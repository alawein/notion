### FMCW Radar for People-Sensing 

Algorized's products are built on millimeter-wave FMCW (Frequency-Modulated Continuous Wave) radar, typically 60–77 GHz. The radar emits frequency-sweeping chirps, mixes the return signal with the transmitted signal to produce a "beat frequency" proportional to target range, and stacks multiple chirps to extract velocity via Doppler FFT. The output is a sparse 3D point cloud — each detection point carries (x, y, z, velocity, SNR) — which serves as the primary input to ML models. 

People-sensing applications include: occupancy detection, activity classification (sitting/standing/walking/falling), gait analysis, breathing rate estimation, and crowd counting. Radar is preferred over camera for privacy-preserving sensing and works through walls, in darkness, and in adverse lighting. 
FMCW Chirp Range FFT Doppler FFT CFAR Point Cloud Micro-Doppler AoA mmWave PointNet++ Q Explain how FMCW radar detects a person's position and velocity. BASIC ▶ Core Answer A chirp sweeps frequency linearly: `f(t) = f_c + (B/T_c)·t `. When reflected from a person at range R, the return signal is delayed by `2R/c `, creating a beat frequency `f_beat = 2BR/(c·T_c) `. A 1D FFT on the ADC samples converts beat frequency → range bins. Stacking M chirps and taking a 2D FFT across them extracts Doppler shift (velocity). A third dimension (angle) comes from spatial FFT across multiple receive antennas. - **Range resolution: **`Δr = c/(2B) `. At B=4 GHz: Δr ≈ 3.75 cm. 
- **Velocity resolution: **`Δv = λ/(2·T_frame) `. Longer frame → finer velocity. 
- **Output: **CFAR thresholding on the range-Doppler-angle cube → sparse point cloud (x, y, z, v, SNR). 
- **Micro-Doppler: **STFT on Doppler dimension reveals breathing (~0.3 Hz), gait (~2–4 Hz), heartbeat (~1 Hz). 
Key Point Frame this as a pipeline: ADC → Range FFT → Doppler FFT → CFAR → Point Cloud → Model. Knowing the chain at signal level is what separates you from a generic ML engineer. Q How do you distinguish a sleeping person from an empty room? MID ▶ Core Answer Sleeping individuals have near-zero macro-motion but measurable micro-Doppler. Chest wall movement from breathing produces a Doppler signature at ~0.2–0.5 Hz. Use a hierarchical classifier: Stage 1 = presence detection via micro-Doppler energy in the slow-time spectrum (high sensitivity, long coherent integration). Stage 2 = activity classifier only when Stage 1 flags presence. - **Data requirement: **Training set must include explicit "stationary occupied" class. Most failures here are data curation failures, not model architecture failures. 
- **Detection trick: **Phase-based breathing detection on static detections — even CFAR misses can show breathing via phase variance across frames. 
- **Avoid: **Classifying purely on point cloud density — sleeping person may generate zero or one CFAR detection. 
Do Not Say "We'd increase the detection sensitivity threshold." Vague. Better: "We'd switch to phase-based micro-Doppler analysis on slow-time frames with extended coherent integration time." Q What are the failure modes when deploying a radar model in a new building? MID ▶ Core Answer - **Multipath reflections: **New wall/furniture layout → spurious point cloud clusters. Fix: empty-room calibration pass → build static clutter map → subtract before CFAR. 
- **Sensor placement shift: **Training assumes ceiling mount; deployed at 1.2m wall height → range/angle distribution mismatch. Fix: normalize to sensor frame; augment training with placement perturbations. 
- **Seasonal drift: **HVAC temperature changes alter noise floor. Fix: track input feature PSI; alert on PSI > 0.2. 
- **Firmware update drift: **DSP parameter changes in firmware alter CFAR output characteristics. Fix: version-lock model binary with firmware version; regression test on known-good captures after each firmware update.

Bias-Variance Regularization Cross-Validation AUC-ROC F1 Calibration Feature Engineering Overfitting Distribution Shift Q Explain the bias-variance tradeoff. How do you navigate it in practice? BASIC ▶ Core Answer The expected MSE of any estimator decomposes as: E[(y - ŷ)²] = Bias² + Variance + Irreducible Noise Bias = error from wrong assumptions (underfit). Variance = sensitivity to training data (overfit). Reducing one increases the other — the tradeoff. - **High bias signals: **High train error, similar test error. Fix: more complex model, more features, less regularization. 
- **High variance signals: **Low train error, high test error. Fix: more data, regularization (L1/L2/dropout), simpler model, ensembling. 
- **In practice: **Plot learning curves (train vs. val loss vs. dataset size). Gap between curves = variance. Both curves high = bias. Use this to decide your next action — not guesswork. 
- **Edge AI context: **Intentional bias (simpler model) is often correct — a 3% accuracy loss is acceptable to hit a 15ms latency budget. 
Q When would you choose F1 over AUC-ROC? What about precision-recall AUC? BASIC ▶ Core Answer - **AUC-ROC: **Threshold-independent measure of rank ordering quality. Good for balanced classes. Misleading for imbalanced: a model predicting all negatives can have AUC ≈ 0.9 on a 10% positive dataset. 
- **Precision-Recall AUC: **Better for imbalanced classes (rare events). Less influenced by true negatives. Use when false negatives are costly (fall detection — missing a fall is worse than a false alarm). 
- **F1: **Single threshold metric, harmonic mean of precision/recall. Use when you need a single number and have a fixed operating threshold. Good for reporting to non-technical stakeholders. 
- **For Algorized: **False positive rate (FPR) independently is the most important metric. A false presence detection wastes energy (HVAC, lighting). Track FPR at your operating threshold explicitly, not buried in AUC curves. 
Trap Answer "I always use F1." Wrong — shows you're not thinking about the problem. Always frame metric choice in terms of the cost of each error type. Q What's the difference between L1 and L2 regularization? When do you use each? BASIC ▶ Core Answer L1: λΣ|wᵢ| → sparse weights (some exactly zero) L2: λΣwᵢ² → small but non-zero weights - **L1 (Lasso): **Produces sparse solutions — useful for feature selection. Geometric: L1 ball has corners that intersect the loss contour at axes (zero weights). Use when you believe few features actually matter. 
- **L2 (Ridge): **Distributes penalty across all weights — more stable, differentiable everywhere. Use when all features contribute somewhat, or when you want numerical stability. 
- **Elastic Net: **L1 + L2 combined. Best of both: sparsity + stability. Standard in many production settings. 
- **Edge AI relevance: **L1 regularization on channel weights in CNNs encourages structured sparsity → easier structured pruning. This is a direct design choice, not an afterthought. 
Q How do you handle class imbalance? What are the failure modes of naive approaches? MID ▶ Core Answer - **Inverse-frequency class weights: **`w_c = n_total / (n_classes · n_c) `. Simple, effective, no data modification. First thing to try. 
- **Focal Loss: **`FL = -αᵢ(1-pᵢ)^γ · log(pᵢ) `. Downweights easy negatives, focuses learning on hard examples. γ=2 standard. Used in RetinaNet; excellent for detection tasks. 
- **SMOTE: **Synthetic oversampling. Failure mode: creates synthetic samples that don't reflect real distribution — can hurt calibration. Use cautiously; validate that synthetics are realistic. 
- **Undersample majority: **Loses information. Only use when majority class is genuinely redundant. 
- **Failure of naive accuracy: **A classifier predicting all "empty" on a 95% empty dataset gets 95% accuracy but zero utility. Always check per-class recall. 
- **For Algorized (fall detection): **Falls are rare (~0.1% of frames). Focal loss + class weighting + strong augmentation of fall examples + separate evaluation of fall recall. 
Q Explain model calibration. What is ECE and why does it matter for a production sensor system? HARD ▶ Core Answer A model is calibrated if its predicted probability matches the empirical frequency of the event. If it says 80% confident, it should be right 80% of the time. ECE = Σ_b (|B_b|/n) · |acc(B_b) − conf(B_b)| - **Modern neural nets are overconfident: **They output 0.99 when they should output 0.7. This is problematic for any downstream decision that uses the confidence score. 
- **Temperature scaling: **Divide logits by learned scalar T before softmax. Post-hoc, no retraining, very effective. T>1 softens probabilities (less confident). 
- **Why it matters for Algorized: **If confidence is used for drift monitoring (flag low-confidence frames for review), or for graceful degradation (fall back to rule-based when confidence < 0.6), miscalibration breaks these systems silently. 
- **Target: **ECE < 0.02 for production. Measure on a calibration set that matches deployment distribution, not the original validation set.

### Neural Networks, CNNs, Transformers, and Point Cloud Models 

Deep learning for this role centers on architectures suitable for radar point cloud data: PointNet/PointNet++ (permutation-invariant point set learning), temporal CNNs (1D/2D conv over frame sequences), and lightweight Transformers for activity classification. You also need solid understanding of backpropagation, batch normalization, gradient flow, and the practical reasons to choose one architecture over another. 

For edge deployment, the architectural constraints are real: no LayerNorm (not INT8-friendly), prefer BatchNorm; avoid dynamic shapes in graph; prefer depthwise separable convolutions; be explicit about FLOPs and parameter count. 
PointNet++ Temporal CNN Transformers BatchNorm Attention Gradient Flow Depthwise Sep. Conv FLOPs Q How does PointNet handle unordered point clouds? Why is permutation invariance important for radar data? MID ▶ Core Answer PointNet applies shared MLP weights independently to each point, then uses a symmetric function (max-pooling) to aggregate into a global feature vector. Because max-pool is invariant to input order, the network is permutation-invariant. f({x₁,...,xₙ}) = γ(MAX{h(xᵢ) : i=1..n}) - **Why radar needs this: **CFAR returns N detections in an arbitrary order. Any order-dependent model (RNN on raw detections, 2D conv on voxel grid) either ignores this or requires expensive preprocessing. 
- **PointNet++ improvement: **Local neighborhood grouping via FPS (Farthest Point Sampling) + ball query. Captures local spatial structure like body parts — critical for activity classification (arm vs. torso velocity). 
- **FPS complexity: **O(N log N) — feasible on-device for N < 512 points per frame. 
- **Practical limitation: **Original PointNet++ doesn't natively handle velocity/SNR as additional features. Extend: treat (x, y, z, v, snr) as 5D point features; the MLP handles it naturally. 
Q Explain batch normalization. What breaks if you replace it with layer normalization for edge deployment? MID ▶ Core Answer BN: x̂ = (x − μ_B) / σ_B, y = γx̂ + β - **BatchNorm: **Normalizes across the batch dimension. At training time: uses batch statistics. At inference: uses running mean/variance (frozen). INT8 quantization fuses BN into the preceding Conv layer — zero additional cost at inference. 
- **LayerNorm: **Normalizes across features within a single sample. Does not fuse with Conv. Requires floating-point normalization at runtime. Quantizing LayerNorm to INT8 causes significant accuracy degradation. 
- **For Algorized's edge deployment: **BatchNorm only. Any architecture using LayerNorm (most Transformers) needs modification before INT8 deployment — either replace with BatchNorm or keep that layer in FP16. 
- **Practical note: **Small batch sizes (batch=1 at edge inference) make BatchNorm variance estimates unstable. Use running statistics from training — never recompute from a batch of 1. 
Common Mistake "LayerNorm and BatchNorm are basically the same." They are not. This distinction is operationally important for INT8 quantization. Say it explicitly. Q Why would you choose a temporal CNN over a Transformer for on-device activity classification? HARD ▶ Core Answer - **FLOPs: **Self-attention is O(L²·d) in sequence length L. For 10 frames, temporal CNN is O(K·L·d) with kernel K — ~10× cheaper than attention. 
- **INT8 compatibility: **CNNs with BatchNorm quantize cleanly. Transformer QK dot-product in INT8 accumulates overflow risk at large sequence lengths — requires careful INT16 accumulation or FP16 for attention weights. 
- **Memory: **Attention scores require O(L²) memory. On Cortex-M this matters; on Cortex-A less so. 
- **When to use Transformer instead: **Long-range temporal dependencies (gait cycle analysis, 30+ frame sequences) where CNN receptive field is insufficient without many layers. Or if a DSP/NPU accelerator handles attention efficiently. 
- **Practical choice for Algorized: **Start with temporal CNN (baseline, fast, deployable). If accuracy gap > 3% on hard cases, try distilling a Transformer teacher into a CNN student. 
03 LLMs & RLHF / DPO Your Differentiator Topic Overview 
### Large Language Models, Alignment, and Your Creative Transfer 

Your LLM/RLHF background is the most unusual and potentially valuable thing about your profile for this role. Most radar data scientists don't know RLHF at all. The key is not to oversell it as "LLM experience" but to concretely explain how the *mechanisms *— preference learning, reward modeling, iterative fine-tuning, evaluation harnesses — transfer to Algorized's problems of annotation quality, environment adaptation, and continuous model improvement. 

SFT = Supervised Fine-Tuning on curated instruction-response pairs. RLHF = Reinforcement Learning from Human Feedback (typically PPO over a reward model). DPO = Direct Preference Optimization (bypasses explicit reward model, trains directly on preference pairs). DPO is simpler, more stable for small datasets, and is what you'd propose at Algorized. 
SFT RLHF DPO Reward Modeling PPO KL Divergence Preference Pairs Policy Gradient Q Explain RLHF at a technical level. Why is it preferred over pure SFT for alignment? MID ▶ Core Answer SFT trains directly on gold-label (prompt, response) pairs. It's limited by the quality and coverage of the demonstration data — if the gold responses don't cover edge cases, the model learns nothing there. RLHF adds a preference signal: given two responses to the same prompt, which is better? - **Step 1 — SFT: **Fine-tune base model on curated demonstrations. Establishes the behavioral prior. 
- **Step 2 — Reward model: **Train a model to predict human preference scores from (prompt, chosen, rejected) triples. Reward model learns the latent quality criterion. 
- **Step 3 — PPO: **Use reward model as a reward signal to fine-tune the policy via proximal policy optimization. KL penalty against SFT reference prevents reward hacking. 
- **Why RLHF > SFT: **SFT learns "what good responses look like." RLHF learns "what makes a response better or worse" — a richer signal that generalizes to unseen cases. 
- **DPO shortcut: **Eliminates Steps 2+3. Rewrites RLHF objective analytically to train directly on preference pairs. No PPO instability, no reward model, simpler to train. Preferred for small datasets and domain adaptation. 
Transfer to Algorized Replace "prompt" with "radar frame context," "response" with "activity classification," and "human preference" with "expert annotator prefers label A over model output B." DPO then becomes a tool for environment-specific adaptation with <200 labeled preference pairs. Q What is reward hacking and how do you prevent it? HARD ▶ Core Answer Reward hacking: the policy learns to exploit weaknesses in the reward model to achieve high reward without achieving the intended behavior. The reward model is an approximation; optimize it too hard and you find its failure modes. - **KL penalty: **Add `-β·KL(π_θ || π_ref) `to the reward. Keeps policy close to the SFT reference. Prevents extreme drift. β is the key hyperparameter. 
- **Reward model ensemble: **Use multiple reward models; only trust agreement. Disagrement regions → high uncertainty → conservative policy gradient update. 
- **Distribution shift monitoring: **Monitor KL divergence between policy outputs and reference distribution over training. If it diverges rapidly, reduce learning rate or increase β. 
- **DPO avoids this differently: **By not explicitly training a reward model, DPO eliminates the reward hacking attack surface. The reference model acts as implicit regularization.

CI/CD OTA Updates PSI KS Test Canary Deployment Rollback Data Lineage Model Registry Q How do you detect concept drift post-deployment with no cloud connectivity? MID ▶ Core Answer - **On-device PSI: **Maintain running histograms of input feature distributions (point cloud density, velocity spread, SNR distribution). Compute Population Stability Index against baseline every N frames. PSI = Σᵢ (Aᵢ - Eᵢ) · ln(Aᵢ/Eᵢ) PSI < 0.1: stable. 0.1–0.2: monitor. >0.2: flag for recalibration. 
- **KS Test: **Two-sample Kolmogorov-Smirnov on selected scalar features. More sensitive to distribution shape changes. Thresholded p-value. 
- **Output confidence monitoring: **Track running mean and variance of model's top-1 confidence score. Systematic drop (e.g., baseline: 0.87 → deployed: 0.71) signals model is uncertain in this environment. 
- **Proxy label drift: **Use physical heuristics as weak supervision — e.g., if occupancy count is inconsistent across overlapping time windows, flag the period for review. 
- **Circular buffer: **Store last 500 feature histograms. O(window) memory. No unbounded growth — critical for embedded Linux.

### Model Compression for C++ Edge Runtimes 

Edge deployment is the core systems challenge at Algorized. Models trained in PyTorch must be compressed (quantization, pruning, distillation) and exported to a C++ runtime (ONNX Runtime, TFLite, TVM, NCNN) for inference on ARM Cortex-A or similar embedded processors. The key constraint is a hard latency budget (typically <20ms P99) and memory budget (often <64 MB RAM). 

INT8 quantization is the primary tool: it provides ~3–4× speedup on SIMD-capable processors (ARM NEON, Qualcomm HVX) by enabling 4-wide dot products vs. scalar FP32. The tradeoff is representational precision — handled by careful calibration and per-channel asymmetric quantization. 
PTQ QAT INT8 Per-Channel ONNX Runtime TFLite Pruning Distillation qnnpack Q Walk me through PTQ vs QAT. When do you use each? MID ▶ Core Answer - **PTQ (Post-Training Quantization): **Quantize after training. Insert activation observers → collect statistics on calibration dataset (500–2000 representative samples) → compute scale/zero-point per tensor or channel → convert. No gradient updates. Fast. First thing to try. 
- **PTQ expected accuracy drop: **0.5–3% for CNNs with BatchNorm. Higher for Transformers (5–8%) due to wider dynamic range in attention scores. 
- **QAT (Quantization-Aware Training): **Simulate quantization noise during training using straight-through estimator (STE) gradients. Model learns to tolerate quantization error. Accuracy drop: typically <0.5%. 
- **Decision rule: **Start with PTQ. If accuracy drop <2%, ship it. If >2%, move to QAT. If still failing, apply mixed-precision (keep sensitive layers in FP16). 
- **Per-channel vs per-tensor: **Per-channel quantization (separate scale per output channel) is almost always better. Per-tensor is faster but degrades depthwise separable conv significantly. 
- **Backend selection: **`qnnpack `for ARM Cortex-A (Android, embedded Linux). `fbgemm `for x86. Choosing wrong backend nullifies the speedup. 
Q You need to cut latency from 50ms to 15ms on Cortex-A55. What's your step-by-step approach? HARD ▶ Core Answer - **Step 1 — Profile first: **Arm Streamline or `perf stat `. Identify which layers consume >80% of latency. Usually the first large conv or any unoptimized depthwise conv. 
- **Step 2 — Layer fusion: **Fuse BN+ReLU+Conv into single kernel (most runtimes do this; verify with ONNX graph inspector). Eliminates memory round-trips. 
- **Step 3 — INT8 quantization: **~3–4× speedup on NEON SIMD. Start here before architectural changes. Validate accuracy drop <2%. 
- **Step 4 — Operator substitution: **Replace standard Conv2D with depthwise separable. FLOPs: `K²·C_in·C_out·H·W → K²·C_in·H·W + C_in·C_out·H·W `. ~9× reduction for K=3. Accuracy hit: 1–3% typical. 
- **Step 5 — Reduce input resolution / receptive field: **Profile whether the model is computing on unnecessarily dense inputs. Subsample point clouds or reduce feature map size. 
- **Step 6 — Architecture NAS or distillation: **If still above budget, distill a smaller architecture with the latency budget as a hard constraint from day 1 of training. 
Don't Skip to Step 6 Immediately proposing "let's distill a smaller model" without profiling first is wrong. You don't know where the time is going. Profile first, always.
