### Artificial Intelligence 

Any computational technique that makes a machine exhibit intelligent behavior. Includes rule-based systems, search algorithms, and learned models. ML is a *subset *of AI. 
Definition 2 
### Machine Learning 

Systems that **learn from data **rather than from explicit rules. Formally: find f* = argmin E[L(f(x), y)] over some function class. DL is a subset of ML. 
Definition 3 
### Deep Learning 

ML using neural networks with many layers, enabling **representation learning **— the model discovers its own features. Not a separate paradigm, but a way to parameterize f. 
The Core Distinction — How to say this in an interview **AI **= the goal (intelligent behavior). **ML **= learning the behavior from data, not coding it. **DL **= using multi-layer neural nets to do that learning, especially when the input is high-dimensional (images, text, audio). You choose DL when you don't want to hand-engineer features and you have enough data. section 02 Neural Networks 
from First Principles A neural network is just a composition of parameterized functions. Understanding this from scratch explains why depth helps, why activations matter, and why backpropagation works. ◉ The Artificial Neuron — One Unit x₁ x₂ x₃ w₁ w₂ w₃ z = Σwᵢxᵢ + b a = σ(z) + b (bias) Activation σ(z) = ReLU GELU / SiLU / tanh output a inputs weights weighted sum + bias non-linearity scalar output z = w₁x₁ + w₂x₂ + … + wₙxₙ + b = **Wx + b **→ output a = σ(z) Why Stack Layers? Why Depth? A single neuron computes a linear function. A single layer of neurons is still a linear map (just multi-dimensional). **Stacking non-linear layers is what gives neural networks their expressive power. **◉ Stacking Layers — How Depth Builds Representations INPUT HIDDEN 1 HIDDEN 2 HIDDEN 3 OUTPUT raw pixels edges + colors shapes + textures object parts class label d=4 ReLU → 5 ReLU → 5 ReLU → 4 ŷ₁ ŷ₂ softmax → FORWARD PASS → Key Insight — Universal Approximation 
### Why Deep Networks Work 

A shallow network with enough neurons can approximate any continuous function (Universal Approximation Theorem). But **deep networks do it exponentially more efficiently **— a function needing 2ⁿ neurons in one layer may need only n×d neurons across d layers. Depth also enables compositional representations: each layer builds on the last. 
Activation Functions — The Non-Linearity ◉ Activation Functions — Shape, Gradient, Use Case ReLU max(0, x) grad=0 grad=1 dead below 0 GELU x · Φ(x) BERT / GPT / ViT Sigmoid 1/(1+e⁻ˣ) saturates! outputs only, binary CE SiLU/Swish x · σ(x) slight dip LLaMA / PaLM Why Not Sigmoid Everywhere? Sigmoid saturates for large |x| — derivative ≈ 0 → vanishing gradients in deep networks. ReLU fixed this (gradient = 1 for x > 0). **GELU and SiLU dominate transformers **because they're smooth (better optimization landscape) and non-monotonic (the slight negative values provide richer gradient signal). section 03 Architecture

CNN assumes nearby pixels are related (local patterns matter). RNN assumes order matters. Transformer assumes **any position can relate to any other **— no locality assumption. This is why transformers need more data but generalize better across diverse tasks. section 04 Why Transformers? The Transformer wasn't invented because RNNs were wrong — they were too slow to scale. Understanding what problem it solved explains every design decision in the architecture. ◉ RNN vs. Transformer — The Core Problem RNN / LSTM — Sequential O(L) steps, cannot parallelize T1 T2 T3 T4 T5 h₁→ h₂→ h₃→ h₄→ T5 must wait for T1→T2→T3→T4→T5 to complete gradient vanishes over long paths • cannot scale on GPU max training sequence: ~500 tokens in practice vs Transformer — Parallel O(1) parallel steps (at cost of O(L²) memory) T1 T2 T3 T4 T5 Every token attends to every other — in ONE parallel step Gradient path: 1 step from any token to any other Scales to 128K+ token contexts with efficient kernels The Key Trade-off 
### O(L) Sequential vs. O(L²) Memory 

RNN: O(L) time steps, O(1) memory per step. But must be sequential — can't parallelize on GPU. Transformer: O(1) "time steps" (parallel), but O(L²) memory for the attention matrix. At L=1000 this is fine; at L=100K you need Flash Attention. The parallelism is what enables training on GPT-4-scale data. 
- **Winner for training: **Transformer — GPU parallelism is the key unlock for scaling. 
- **Winner for streaming inference: **LSTM/Mamba — process one token, maintain state, no O(L²). 
- **Flash Attention: **Computes the same attention matrix without materializing it in HBM — O(√L) memory, same mathematical result, 2–4× faster.

Diagonal = attending to self Why √dₖ Scaling? For random Q,K vectors, the dot product variance grows as dₖ . At dₖ=64 , scores blow up → softmax saturates → gradient ≈ 0. Dividing by √dₖ keeps variance ≈ 1. Multi-Head Attention Key Mechanism 
### Why Multiple Heads? 

A single attention head produces a single attention pattern per layer. Multi-head attention runs h parallel attention heads, each with its own Q/K/V projections. Each head can specialize: one may track syntax, another coreference, another sentence structure. 
- **Each head dimension: **dₖ = d_model / h. Same total compute, but different "views" of the input. 
- **Output: **Concatenate all head outputs → linear projection Wᴼ → d_model. 
- **GQA (LLaMA-2/3): **Grouped Query Attention — share K,V across groups of Q heads. Reduces KV cache size ÷ n_groups. Best of both MHA and MQA. 
- **Typical: LLaMA-3-8B: **32 Q heads, 8 KV heads (GQA). d_model=4096, d_head=128. 
section 06 Full Transformer Block The transformer block is the repeating unit. Understanding what each component does — and in what order — is a prerequisite for any architecture question. ◉ Decoder-Only Transformer Block (LLaMA/GPT-4 style — Pre-Norm) Token Embeddings xᵒ = Embed(tokens) + PositionalEnc(pos) × N layers RMSNorm / LayerNorm (pre-norm) normalize BEFORE sublayer — more stable at scale Multi-Head Self-Attention Wᵠ → Q "what I seek" Wᵏ → K "what I have" Wᵛ → V "what I give" softmax (QKᵀ/√d)·V residual x + x = x + attn_output RMSNorm (second) normalize before FFN residual x Feed-Forward Network (FFN) Linear(d_model → 4·d_model) → SiLU/GELU → Linear(4·d → d) SwiGLU variant: gate(x)·up(x) — used in LLaMA / PaLM + x = x + ffn_output x_out ∈ ℝᴸˣᵈ input to next block (or to LM head) Pre-Norm vs Post-Norm 
### Why Pre-Norm Wins 

**Post-Norm **(original paper): LayerNorm(x + Sublayer(x)). Unstable at large scale — gradient depends on sublayer output which can vary wildly early in training. 

**Pre-Norm **(modern): x + Sublayer(LayerNorm(x)). The residual stream x is never normalized out — gradients flow cleanly. Enables training without warmup in some regimes. Used in GPT-3, LLaMA, all modern LLMs. 
The FFN — What Does it Do? 
### FFN as Key-Value Memory 

Mechanistic interpretability research (Geva et al., 2021) shows FFN layers act as memory: the first linear layer "keys" activate for specific input patterns; the second "values" retrieve stored information. This is why LLMs can recall facts without retrieval — knowledge is stored in FFN weights. 
- **d_ff = 4×d_model: **4× expansion ratio is standard. SwiGLU uses 2/3 × 4 = 8/3 for equivalent capacity. 
- **Share parameters: **Some architectures (MoE) route each token to different FFN experts — capacity without compute. 
section 07 The Training Loop Training is a cycle: forward pass computes predictions, loss measures error, backward pass computes gradients, optimizer updates weights. Everything else is scaffolding around this core loop. ◉ The Training Cycle — One Step TRAINING STEP ① Sample Batch (x_batch, y_batch) from DataLoader shuffle=True each epoch ② Forward Pass ŷ = model(x) with autocast() for FP16/BF16 ③ Compute Loss L = CE(ŷ, y) MSE/BCE/focal/etc. ④ Backward Pass loss.backward() ∂L/∂w via chain rule ⑤ Clip + Update clip_grad_norm_ (max 1.0) optimizer.step() repeat for every batch 1 pass over full dataset = 1 epoch ● Critical Concept 
### Backpropagation — What it Actually Is 

Backpropagation is **reverse-mode automatic differentiation **applied to the computation graph of the forward pass. It uses the chain rule to efficiently compute ∂L/∂wᵢ for every parameter in O(forward pass cost) time. 
- **Chain rule: **∂L/∂w₁ = ∂L/∂a_n · ∂a_n/∂a_{n-1} · … · ∂a_2/∂a_1 · ∂a_1/∂w₁ 
- **Why vanishing gradients happen: **Each term in this product is a Jacobian. If each is <1 (sigmoid saturated), product → 0 exponentially fast with depth. 
- **Why residual connections fix it: **x + F(x) has gradient 1 + ∂F/∂x. The "+1" term provides a direct gradient highway through every layer. 
- **PyTorch autograd: **Builds a dynamic computation graph during the forward pass. `loss.backward() `traverses it in reverse. `.detach() `cuts gradients. `requires_grad=False `freezes parameters.

Sample Answers These are the questions that actually come up at the Senior DS level. The answers below are timed to ~90 seconds — confident, concrete, and technically precise. ARCH What's the difference between AI, ML, and Deep Learning? How do they relate? BASIC ▶ Sample Answer (~60s) AI is the broadest category — any technique making machines appear intelligent, including rule-based expert systems and search algorithms. Machine learning is a subset of AI where the system learns patterns from data rather than following hand-crafted rules. Deep learning is a subset of ML that specifically uses multi-layer neural networks to learn hierarchical representations, which works especially well for high-dimensional data like images, text, and audio. - **The practical distinction: **You choose DL over classical ML when you can't hand-engineer good features — for example, you can't tell someone "this pixel pattern means cat." DL learns those features automatically. 
- **When NOT to use DL: **Tabular data with <10K rows — gradient boosting (XGBoost/LightGBM) almost always wins. DL's sample efficiency is poor for small, structured datasets. 
- **Bridge to your background: **In computational materials science, DFT gives you quantum mechanical features that are hard to engineer — that's exactly the regime where a surrogate ML model (even classical RF) provides 60–70% speedup. 
TRANSFORMER Explain how the Transformer works. Why did it replace RNNs? MID ▶ Sample Answer (~90s) The Transformer processes all tokens in a sequence in parallel, using self-attention to compute relationships between any pair of positions in a single step. Each token produces three vectors — query, key, value — via learned projections. The attention score between two tokens is the dot product of query and key, scaled by √dₖ and put through softmax. The output is a weighted average of all value vectors, where the weights are those attention scores. - **Why it replaced RNNs: **RNNs process tokens sequentially — T5 has to wait for T1→T2→T3→T4. This makes them unparallelizable and causes vanishing gradients over long sequences. Transformers compute all attention in one parallel matrix multiply, so the gradient path between any two tokens is length 1, not length L. 
- **The trade-off: **RNNs use O(1) memory per step and O(L) time. Transformers use O(L²) memory for the attention matrix but O(1) "sequential" steps. Flash Attention reduces the memory to O(√L) without changing the math. 
- **Why decoder-only won: **Next-token prediction scales cleanly with data and compute. The same model handles generation, classification, and reasoning through prompting. BERT's ceiling was ~340M params; decoder models scale to 1T+. 
TRAINING Walk me through how you'd fine-tune an LLM for a custom task from scratch. HARD ▶ Sample Answer — Step-by-Step (~90s) - **1. Define the task precisely: **What's the input/output format? What metric measures success? Don't start until you can answer both. 
- **2. Choose base model: **Consider license (Mistral = Apache 2.0, LLaMA = research), context length needed, and size. For most tasks, 7B is the sweet spot — fits on 1 GPU with QLoRA, good capability. 
- **3. Curate training data: **Minimum ~1K instruction-response pairs for SFT. Format: ChatML. Deduplicate with MinHash. Human-review 50 random samples before training. 
- **4. Configure LoRA: **r=16, lora_alpha=32, target_modules=["q_proj","k_proj","v_proj","o_proj"]. Load model in 4-bit NF4 if GPU-constrained (QLoRA). 
- **5. Train with TRL SFTTrainer: **lr=2e-4, batch=4, grad_accum=4 (effective batch=16), cosine schedule, warmup_ratio=0.05, 3 epochs. Log to W&B. 
- **6. Evaluate honestly: **Human evaluation on 100 held-out examples. Task-specific metric. Compare vs. zero-shot baseline. Is the gap worth the cost? 
- **7. Merge and serve: **merge_and_unload() on LoRA → vLLM for serving → canary 1% rollout → monitor latency and output distribution. 
TRAINING What is RLHF? How does DPO differ and when would you prefer it? HARD ▶ Sample Answer — 3-Stage vs Direct RLHF (Reinforcement Learning from Human Feedback) aligns a language model to human preferences in three stages: SFT on instruction data, then train a reward model on (prompt, chosen, rejected) triples using a Bradley-Terry model, then fine-tune the policy with PPO to maximize the reward model's score subject to a KL penalty against the SFT reference model. - **DPO (Direct Preference Optimization): **Skips the explicit reward model entirely. It shows that the optimal RLHF policy has a closed-form relationship to the preference data. You train directly on (chosen, rejected) pairs with the DPO loss: −log σ(β·(log π_θ(y_w|x)/π_ref − log π_θ(y_l|x)/π_ref)). 
- **When to prefer DPO: **You don't need a separate reward model (simpler pipeline, less memory). Training is stable — no RL instability. Approximately equivalent quality to PPO. My Turing work used this pattern for its preference alignment pipeline. 
- **When PPO is better: **Online RL — generating new responses and scoring them dynamically, rather than learning from fixed offline preference pairs. Better for tasks where exploration matters (e.g., coding with unit test feedback). 
- **β parameter: **Controls the KL penalty. High β → model stays close to SFT reference (conservative). Low β → more alignment but risks forgetting general capability. 
SYSTEM Your model works great in testing but degrades after 2 weeks in production. How do you debug it? HARD ▶ Senior Answer — Systematic Diagnosis This is almost always distribution shift — production data has drifted from training data. The diagnostic protocol has a fixed order: don't assume, measure first. - **Step 1: Check if performance actually dropped. **Calculate the metric on a labeled sample of recent production examples. Sometimes the "degradation" is a noisy signal from a different monitoring tool. 
- **Step 2: Compute PSI on all input features. **Population Stability Index >0.2 on any feature means significant distribution shift. Identify which features shifted — this points to the source (new user population, changed upstream data pipeline, seasonality). 
- **Step 3: Check train-serve skew. **Log both raw inputs and preprocessed features at inference. Compare preprocessing output distribution against what training saw. Silent skew is the most common production bug. 
- **Step 4: Rollback first, diagnose second. **Don't retrain speculatively. Roll back to last known-good model to stop the bleeding, then diagnose with a clear head. 
- **Step 5: Fix and validate. **If shift: retrain on recent data, add temporal mixing to prevent future degradation. If preprocessing: fix the pipeline, validate both directions. Add monitoring alerts for PSI >0.1 (warning) and >0.2 (retrain trigger). 
Bridge to your background "In my DFT pipeline work, I ran into this pattern — a batch optimization reordered output indices silently. My fix was to add checksum validation on every pipeline stage output, which is the same principle as monitoring feature distributions in production ML." OPTIM What is the vanishing gradient problem and how does the Transformer solve it? MID ▶ Sample Answer Backpropagation computes gradients via the chain rule — the gradient at layer 1 is the product of Jacobians at every layer from the output back to layer 1. If each Jacobian has eigenvalues <1 (which happens when activations saturate, like sigmoid), this product shrinks exponentially with depth. In a 20-layer sigmoid network, gradients at layer 1 are ≈ 0 — those weights don't learn. - **How ReLU helped: **Gradient is 1 for x>0, so it doesn't inherently shrink. But dead neurons (output always 0) are still a problem for large negative inputs. 
- **How residual connections solve it: **x + F(x) has gradient 1 + ∂F/∂x. The +1 is a direct highway from loss to any layer regardless of depth. Even if ∂F/∂x ≈ 0, gradient still flows. 
- **How Transformer specifically handles it: **Pre-LayerNorm (normalize before sublayer) keeps activation statistics stable. Residual connections after both attention and FFN. GELU/SiLU instead of sigmoid. These together let 100+ layer transformers train stably. 
DATA What is data leakage and how do you prevent it? MID ▶ Sample Answer Data leakage is when information from the test set (or future information) leaks into the training process, making validation metrics appear better than real-world performance will be. It's the most common way to produce an "impressive" model that fails in production. - **Feature leakage: **Using a feature that's only available after the prediction is needed (e.g., using discharge date to predict readmission risk — discharge date isn't known when you need the prediction). Fix: audit every feature for availability at inference time. 
- **Preprocessing leakage: **Fitting a StandardScaler on the full dataset before splitting, then "splitting." The scaler has seen test data. Fix: always fit preprocessors on training data only, then transform test. 
- **Temporal leakage: **Randomly splitting a time-series dataset so future records appear in training. Fix: always split by time. Train on past, validate on future. 
- **Group leakage: **Same patient, user, or document appears in both train and test. The model learns the identity, not the pattern. Fix: GroupKFold or group-stratified split. 
- **Detection: **Suspiciously high accuracy is the main signal. AUC=0.99 on a hard problem → check for leakage before celebrating.

COMPUTE How do you train a 70B parameter model when it doesn't fit on a single GPU? HARD ▶ Sample Answer — Parallelism Strategies A 70B model in BF16 needs 140GB — that's more than a single A100 80GB. There are three main parallelism strategies, and in practice you combine them. - **Data Parallelism (DDP): **Each GPU holds the full model; different data batches per GPU. Gradients are averaged across GPUs via all-reduce. Works only if the model fits on one GPU. For 70B: doesn't work alone. 
- **Tensor Parallelism (TP): **Split individual weight matrices across GPUs column/row-wise. Each GPU computes part of each layer. Requires fast inter-GPU bandwidth (NVLink). Megatron-LM does this. 8-way TP on 8×A100 = each GPU holds 70B/8 = ~8.75B params. 
- **Pipeline Parallelism (PP): **Split the model vertically — GPU 0 holds layers 1–20, GPU 1 holds 21–40, etc. Data flows through as a pipeline. Bubble inefficiency at boundaries. 
- **DeepSpeed ZeRO-3: **Shards optimizer states, gradients, AND parameters across all GPUs. Each GPU holds only 1/N of each. Allows training any size model on N GPUs. Most practical for fine-tuning. 
- **Practical recipe: **QLoRA for fine-tuning (4-bit base + BF16 adapters) on 2×A100 40GB. For pretraining: ZeRO-3 + gradient checkpointing + BF16 + flash attention across 16+ GPUs. 
EVAL How do you evaluate an LLM? What metrics do you use and why is this hard? HARD ▶ Sample Answer — The Evaluation Problem Evaluating LLMs is genuinely unsolved. The fundamental problem is that we want to measure "helpfulness" and "correctness," neither of which is a single number. You need a layered evaluation approach. - **Automated benchmarks: **MMLU (multi-task knowledge), HellaSwag (commonsense), HumanEval (code), GSM8K (math reasoning), TruthfulQA (factuality). These are multiple-choice or exact-match — fast, cheap, reproducible. Problem: models overfit to benchmark phrasing; doesn't measure real-world usefulness. 
- **LLM-as-judge: **Use GPT-4 to score outputs on criteria (coherence, accuracy, helpfulness) from 1–5. MT-Bench does this. Much more flexible than exact-match. Problem: LLM judges have systematic biases (prefer longer outputs, prefer their own style). 
- **Human evaluation: **The ground truth. A/B preference ratings between model outputs. Expensive and slow but necessary before deployment. Use for final gating, not for iteration. 
- **Task-specific metrics: **For classification: accuracy, F1, AUC-PR. For generation: ROUGE (n-gram overlap), BERTScore (embedding similarity). For summarization: faithfulness scores. For code: pass@k (does generated code pass unit tests?). 
- **Why it's hard: **A model can score well on benchmarks but fail at your specific use case. Always evaluate on a held-out set of real task examples that mirrors production distribution. 
ARCH What is a Mixture of Experts (MoE) and why do models like Mixtral use it? MID ▶ Sample Answer In a standard transformer, every token passes through every FFN neuron on every forward pass. Mixture of Experts replaces each FFN layer with N expert FFNs plus a router, where each token is only sent to 2 (or k) of the N experts. - **Why this is powerful: **Mixtral-8x7B has 8 experts per layer, uses top-2 routing. Total parameters: ~47B. But active parameters per token: ~13B (only 2/8 experts activate). You get near-47B model quality at ~13B compute cost per forward pass. 
- **The router: **A small learned linear layer takes the token embedding and outputs a probability over experts. Typically softmax scores, select top-k. The router output weights are multiplied into the expert outputs before summing. 
- **Load balancing: **Without explicit regularization, the router collapses — it learns to always use the same 1–2 experts. An auxiliary load-balancing loss ensures roughly equal expert utilization during training. 
- **Trade-off: **MoE is more memory-intensive (must store all experts) but faster at inference (fewer active FLOPs). Requires expert parallelism across GPUs in production — each GPU holds a subset of experts. 
- **Models: **Mixtral-8x7B, Mixtral-8x22B, GPT-4 (reportedly), Google's Switch Transformer. Increasingly common at scale because parameter efficiency improves faster than FLOP efficiency. 
DESIGN Design a system to detect AI-generated text at scale in a production pipeline. SYSTEM DESIGN ▶ Senior System Design Answer This is a system design question wearing ML clothes. Start by clarifying constraints: What's the volume? What's the acceptable false positive rate? What's the latency budget? Then design accordingly. - **Problem framing: **Binary classification (AI vs human) but extremely hard because: (1) LLM outputs overlap with human writing, (2) easy paraphrasing bypass, (3) distribution shifts as models improve. Set expectations: >95% accuracy is unrealistic for short texts. 
- **Signal 1 — Perplexity: **LLMs generate high-probability text. Score input with a reference LLM (e.g., GPT-2). Low perplexity = LLM-like. Fast, no training needed. But: high-quality human writing also has low perplexity. 
- **Signal 2 — Trained classifier: **Fine-tune RoBERTa or similar on (human, AI) pairs. OpenAI's detector uses this approach. Faster inference, higher accuracy for in-distribution models. Degrades for new model families. 
- **Signal 3 — Watermarking (if you control generation): **During generation, softly bias token selection toward a pre-committed pseudorandom pattern. Detector checks for pattern. Zero false positives, fully robust to paraphrasing (provably). Only works if you control the generator. 
- **Production architecture: **Route traffic to fast perplexity check first (100ms). High-confidence clear cases exit early. Ambiguous cases go to classifier (500ms). Flag for human review if both are uncertain. Log everything — retrain classifier quarterly as new models emerge. 
- **The honest answer: **This is an arms race. For high-stakes decisions (academic integrity), supplement with behavioral signals (typing patterns, edit history) rather than relying on content alone.
